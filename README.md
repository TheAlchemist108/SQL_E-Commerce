# SQL E-Commerce Analytics Portfolio

A SQL portfolio project built around a realistic, messy e-commerce dataset:
raw data is generated, cleaned through an ETL pipeline into a normalized
schema, then queried with a set of business-question-driven SQL scripts
covering CTEs, window functions, correlated subqueries, and conditional
aggregation.

## Project structure

```
data/
  raw_online_retail.csv   generated flat export (one row per invoice line)
  ecommerce.db            SQLite database built from the cleaned data
sql/
  schema/schema.sql       normalized 4-table schema + indexes
  queries/                8 business-question SQL scripts
scripts/
  generate_data.py        generates the synthetic raw CSV
  etl_load.py              cleans the CSV and loads it into SQLite
  run_queries.py            runs every query and writes docs/query_results.md
docs/
  query_results.md        each query's SQL + real output, auto-generated
```

## About the data

The dataset is synthetic, generated by [`scripts/generate_data.py`](scripts/generate_data.py)
using only Python's standard library (no external download). Its shape and
data-quality problems are modeled directly on the well-known UCI "Online
Retail II" dataset (a UK-based online gift retailer, Dec 2009-Dec 2011) — the
de facto standard practice dataset for SQL/ETL portfolios. A generator was
used instead of bundling the original file so the project is fully
self-contained and reproducible (`random.seed(42)`).

Intentional messiness injected into the raw export, documented in the
generator itself:

1. ~6% of rows have a missing `CustomerID` → guest checkouts
2. Cancellations are separate invoices with `InvoiceNo` prefixed `C` and negative `Quantity`
3. Non-product `StockCode` rows for postage/fees (`POST`, `DOT`, `M`, `BANK CHARGES`, `AMAZONFEE`)
4. Inconsistent `Description` casing/whitespace for the same `StockCode` across rows
5. Inconsistently formatted `Country` values (`"UK"`, `"U.K."`, `"  united kingdom"`)
6. A handful of exact duplicate rows (double-submitted order lines)
7. A handful of `UnitPrice <= 0` rows (data-entry errors / free samples)
8. Whitespace-padded `InvoiceNo` / `StockCode` on a subset of rows

## ETL and schema design

[`scripts/etl_load.py`](scripts/etl_load.py) cleans the raw CSV and loads it into
`data/ecommerce.db` per a specific rule for each issue above (strip
whitespace, canonicalize country names, map missing customers to a synthetic
`customer_id = -1` "guest" row, mark cancellations, pick the most frequent
description per product code, flag non-product lines, de-duplicate exact
repeats, and normalize dates to ISO 8601).

The flat file is normalized into four tables ([`sql/schema/schema.sql`](sql/schema/schema.sql))
so the project supports real joins, subqueries, and window functions instead
of grouping a single flat table:

| Table | Grain |
|---|---|
| `customers` | one row per customer (`-1` reserved for guests) |
| `products` | one row per `stock_code`, with one canonical description |
| `invoices` | one row per `invoice_no` (order/cancellation header) |
| `invoice_lines` | one row per line item, FK to `invoices` and `products` |

## Queries

Each script in [`sql/queries/`](sql/queries) starts with a comment block stating the
business question and the SQL technique used, so the "why" is documented
alongside the "how":

| # | Question | Technique |
|---|---|---|
| [00](sql/queries/00_data_quality_checks.sql) | How dirty is the loaded data? | `UNION ALL`, scalar subqueries |
| [01](sql/queries/01_monthly_revenue_running_total.sql) | Monthly revenue and running total | CTE + window `SUM() OVER` |
| [02](sql/queries/02_month_over_month_growth.sql) | Month-over-month growth % | `LAG()` window function |
| [03](sql/queries/03_top_products_by_country.sql) | Top 2 products per country | CTE + `RANK() PARTITION BY` |
| [04](sql/queries/04_customer_rfm_segmentation.sql) | RFM customer segmentation | `NTILE()` quintile scoring, `CASE` |
| [05](sql/queries/05_repeat_purchase_cohort_retention.sql) | Cohort repeat-purchase retention | self-referencing joins, conditional aggregation |
| [06](sql/queries/06_customers_no_purchase_last_quarter.sql) | Lapsed customers (no Q4 order) | correlated `NOT EXISTS` |
| [07](sql/queries/07_orders_above_customer_avg.sql) | Orders 2x a customer's own average | correlated subquery in `HAVING` |
| [08](sql/queries/08_cancellation_rate_by_country.sql) | Cancellation rate by country | `COUNT ... FILTER`, scalar subquery benchmark |

Full SQL and real output for every query: [`docs/query_results.md`](docs/query_results.md).

### Sample: monthly revenue running total

```sql
WITH monthly AS (
    SELECT
        strftime('%Y-%m', i.invoice_date) AS month,
        ROUND(SUM(il.line_revenue), 2)     AS monthly_revenue
    FROM invoice_lines il
    JOIN invoices i  ON i.invoice_no = il.invoice_no
    JOIN products p  ON p.stock_code = il.stock_code
    WHERE p.is_product = 1
    GROUP BY month
)
SELECT
    month,
    monthly_revenue,
    ROUND(SUM(monthly_revenue) OVER (ORDER BY month), 2) AS running_total_revenue
FROM monthly
ORDER BY month;
```

| month | monthly_revenue | running_total_revenue |
|---|---|---|
| 2023-01 | 18402.55 | 18402.55 |
| 2023-02 | 17838.20 | 36240.75 |
| 2023-03 | 17396.56 | 53637.31 |
| ... | ... | ... |

## Running it yourself

```bash
python scripts/generate_data.py   # writes data/raw_online_retail.csv
python scripts/etl_load.py        # builds data/ecommerce.db from schema + CSV
python scripts/run_queries.py     # runs all queries, writes docs/query_results.md
```

Requires Python 3 (standard library only — `csv`, `sqlite3`, `random`,
`datetime`). No external dependencies.

You can also open `data/ecommerce.db` in any SQLite client and run the
scripts in `sql/queries/` directly.
